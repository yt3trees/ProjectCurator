  特におすすめ

  1. "Where Was I?" コンテキスト復帰ブリーフィング

  プロジェクトを数日〜数週間ぶりに開いたとき、「前回からの差分」をAIが要約する。focus_history、git
  log、完了タスク、新しいdecision_logを時間差に応じて合成し、「3日間で何が変わったか、今何に注意すべきか」を提示。研究に
  よるとコンテキストスイッチで認知能力の20%が失われ、復帰に23分かかる。これを短縮する。
  - 既存のWhatsNextとの違い: WhatsNextは「今何をすべきか」、これは「不在中に何が起きたか」
  - データは全て既存。focus_history/の日付ベース差分 + git log + Asana完了タスク

  2. Scope Drift Detection / フォーカスのズレ検出

  project_summary.mdの目標と、直近2週間のfocus_historyの実際の作業内容をLLMに比較させ、「宣言した目標」と「実際にやって
  いること」の乖離を検出する。「このプロジェクトのゴールはAPI層構築だが、直近はUIの微調整ばかり。意図的?」のように警告。
  - focus_history/の時系列データを持っているProjectCuratorならではの機能
  - 個人開発者には外部からの軌道修正がないので、AIがその役割を果たす

  3. リアルタイム意思決定キャプチャ (Editor Save Hook)

  current_focus.mdを編集して保存したとき、DiffPlexで差分を検出し、構造的な変更(優先度の入れ替え、項目削除、新規目標追加)
  があれば「なぜこの変更をしましたか? Decision Logに記録しますか?」とプロンプト表示。既存のDecisionLogGeneratorServiceは
  バッチ検出だが、これは意思決定の瞬間にリアルタイムで捕捉する。
  - Addy Osmaniのブログでも「最も価値のあるログは決定の瞬間に記録されたもの」と指摘されている
  - EditorのDiff基盤が既にあるので、保存イベントにフックするだけ

  4. Cross-Project Insight / プロジェクト横断インサイト

  全プロジェクトのcurrent_focus、tensions、decision_logをLLMに横断分析させ、非自明な繋がりを検出する。「プロジェクトAの
  新認証モジュールはプロジェクトBで再利用できそう」「プロジェクトCとDの両方にインフラ系の期限切れタスクがある、まとめて
  対処しては?」
  - WhatsNextが「次に何をするか」なら、これは「見落としている関係性」を発見する
  - プロンプト設計が難しいが、マルチプロジェクト管理者ならではの価値

  5. Git Commit Narrative / コミット履歴ナラティブ

  TimelineやDashboardから「今週の変更サマリー」ボタンで、プロジェクト配下の全repoのgit
  logをLLMに渡し、コミット一覧ではなく人間が読める物語として変更内容を生成する。「今週の主な作業はDBのSQLiteからPostgreS
  QLへの移行。移行テスト中に発見された認証フローのエッジケースを2件修正。」
  - 週次レポートや上司への報告にそのまま使える
  - ProjectDiscoveryServiceが既にrepoの場所を知っている

  ---
  野心的だが面白いもの

  6. Natural Language Project Q&A / チャットで質問

  チャットパネルで「プロジェクトXの認証に関する決定は?」「2週間以上触ってないプロジェクトは?」のような自然言語クエリに回
  答。コンテキストファイル群をRAG的に検索してLLMが回答する。
  - 実装コスト高(ツール呼び出しorファイル選択ロジックが必要)だが、プロジェクトが増えるほど価値が増す

  7. Tension Resolver / 課題解決アドバイザー

  tensions.mdの項目を選択すると、project_summary + decision_log +
  current_focusをコンテキストに、LLMが「選択肢A/B/C、それぞれのリスク、推奨」を構造的に提示。そのままDecision Log
  Generatorに繋げて記録できるワークフロー。

  A. ソクラテス式「考える壁打ち」モード

  AIが答えを出さず、あえて問いを返し続けるモード。workstreamを開いて「実装方針に迷ってる」と書くと、AIは「その場合、Xが
  空だったらどうなります?」「Yの可能性は排除できましたか?」のように問いかけ続ける。答えではなく自分の思考の矛盾や盲点を
  炙り出す。最後に「では結論を記録しますか?」とDecision Logに繋ぐ。

  B. グローバルキャプチャ + AI自動振り分け

  ホットキー一発でどこからでもフリーテキストを入力。AIが「これはタスク(Asana)か、意思決定(Decision
  Log)か、テンション(tensions.md)か、フォーカス更新か」を自動判定してルーティング。ナビゲーションなし、ファイルを開かな
  くていい。「〇〇の実装をやらなきゃ → Asana追加」「〇〇で迷ってる → tensions.md」「今日は〇〇に集中する →
  focus更新の起点」などに繋がる。

  C. AI-to-AI コンテキストエクスポート

  外部でClaude/ChatGPTとプロジェクトについて相談したいとき、そのプロジェクトの状態を一つのペーストできるブロブに圧縮して
  生成する。focus + summary + tensions +
  直近decision_logsを構造化テキストに詰め込み、「別のAIセッションを開いてこれを貼れば続きから話せる」状態を作る。(arXiv
  2025の"Lore Protocol"や"State Export Hack"が同じ発想)

  ---
  開発者として刺さるもの

  D. コミットを「意思決定記録」として書くアシスタント

  git diffを読んで「何をしたか」は自動生成するのは当然として、「なぜ他の方法ではなくこれにしたか」「却下した代替案は何か
  」「次にここを触る人への警告は?」 を入力するダイアログを表示。この構造化コミットメッセージがそのままDecision
  Logになる。2025年3月のarXiv論文(Lore Protocol)がまさにこれを提唱している。

  E. Decision Log セマンティック検索

  「前に似たような設計問題を解決したとき、何を決めたっけ?」をキーワードではなく問題の形で検索する。「循環依存が出たとき
  」「DBのスキーマ設計で迷ったとき」のようなクエリで、全プロジェクトの全decision_logを横断して関連する過去の意思決定を引
  っ張ってくる。Khoj(OSS)がObsidianでやっているのと同じことを、decision_log特化で実装する。

  ---
  地味だけど日常的に超便利

  F. 終業ルーティン「Done for Today」

  作業終了ボタンを押すと、今日やったこと(Asana完了 + git commits + focus変更)をAIが整理して:
  1. 変更があったfocus文書を自動更新の候補として提示
  2. 明日の最初にやること(Tomorrow's First Task)を提案
  3. 翌朝のStandupに「昨日やったこと」の精度の高いベースを生成

  「仕事を切る」ルーティンを意識的に作ることで、翌朝の"Where Was I?"問題が大幅に軽減される。

  G. Prompt-to-Focus: 自然言語でフォーカス全書き換え

  「今週はXに集中することにした、なぜなら〜」と一文書くだけで、AIが現在のcurrent_focus.mdの構造を維持しながら、その方針
  転換を反映した形に全体を書き直す。いまのUpdate
  Focusは「Asanaタスクをベースに更新」だが、これは「人間が方針を決めて、AIが文書に反映する」方向の逆引きフロー。

  ---
  個人的には B(グローバルキャプチャ) と D(コミット→意思決定記録) が特に面白いと思います。Bはアプリのインタラクションを根
  本から変える可能性があり、Dは開発者として毎日のコミット作業と意思決定記録が一体化する。

 AI機能アイデア集

  1. Context Switch Briefing (復帰ブリーフィング)

  しばらく触れていなかったプロジェクトを Dashboard/Editor
  で開いたとき、「以前のあなた、ここまでやってましたよ」と要約してくれる機能。

  - トリガー: Focus が N日以上古いプロジェクトをクリックしたとき、ポップアップで提案
  - 入力データ: 直近の focus_history/ diff、decision_log/ 最新数件、未完了Asanaタスクの変化、git の未コミット変更
  - 出力: 「前回ここまで進んでいた / その後こうなった / 今日再開するならここから」の3段構成ブリーフィング
  - 実装: ProjectDiscoveryService の FocusAge を見て閾値判定、FocusUpdateService と同様のパターンでプロンプト構築

  復帰コストが一番高い「あれ、何やってたっけ」を解消する。What's Nextが全体俯瞰なのに対し、こちらは1プロジェクト深掘り。

  ---
  2. Tension Synthesizer (テンション棚卸し)

  tensions.md に溜まった項目を定期的に「まだ生きてる？もう解決した？」と棚卸しする機能。

  - トリガー: Editor で tensions.md を開いたとき、またはCommand Paletteから手動実行
  - 入力データ: tensions.md 全項目 + 直近の decision_log/ + current_focus.md + focus_history/ diff
  - LLMの仕事:
    - 各tension に対し「Resolved(この決定で解消済み) / Active(まだ未解決) /
  Stale(3週間触れてない、エスカレーション推奨)」を判定
    - Resolved なものは該当 decision_log へのリンク付きで削除提案
    - 関連する tension をクラスタリングして「これらは同じ論点では？」と提示
  - 出力: diff形式で確認 → 承認して tensions.md を更新

  Decision Logの「resolved_tension」機能の逆引き版。放置テンションの蓄積を防ぐ。

  ---
  3. Weekly Digest (週次ダイジェスト)

  金曜日(または設定日)に、全プロジェクト横断の「今週の振り返り + 来週の見通し」を自動生成。

  - トリガー: 設定した曜日にアプリ起動時に生成(Standup と同じパターン)、またはCommand Paletteから手動
  - 入力データ: 今週の focus_history/ 全diff、decision_log/ 全件、完了Asanaタスク、Today Queueの来週分
  - 出力:
    - This Week: プロジェクトごとに「やったこと」を自然言語で要約
    - Decisions Made: 今週の意思決定を横断一覧
    - Drift Report: 月曜のfocusと金曜のfocusを比較して「計画 vs 実態」のズレを可視化
    - Next Week Outlook: 来週期限のタスク + 未解決tensionから予測
  - 保存先: {ObsidianVaultRoot}/weekly/YYYY-WNN_digest.md

  既存の Standup
  が「今日」のスナップショットなのに対し、こちらは「1週間」の物語。上司への報告や振り返りにそのまま使える。

  ---
  4. Decision Impact Tracker (決定追跡)

  過去の decision_log に書かれた「Revisit Trigger」や「Risk」が実際にどうなったかを追跡する機能。

  - トリガー: Command Palette、または decision_log が一定数(例: 10件)を超えたとき Dashboard に通知バッジ
  - 入力データ: 過去の decision_log/ 全件(特に Risk, Revisit Trigger セクション) + 現在の current_focus.md + tensions.md
  - LLMの仕事:
    - 各 Revisit Trigger の条件を現在の状況と照合
    - 「この決定、見直し時期かも」をリスト化
    - Riskが現実化していないか tensions と突合
  - 出力: 「要レビュー」「問題なし」「情報不足」の3分類でリスト表示、該当ファイルへのリンク付き

  Decision Logを「書いて終わり」にしない仕組み。意思決定の品質が上がる。

  ---
  5. Project Weather Report (プロジェクト天気予報)

  Dashboard の各プロジェクトカードに、1行の「天気」を表示する機能。

  - トリガー: Dashboard リフレッシュ時(自動更新サイクルに乗せる or 手動ボタン)
  - 入力データ: 各プロジェクトの FocusAge, SummaryAge, DecisionLogCount, 未コミット数, TodayQueue のoverdue件数,
  tension件数
  - LLMの仕事: メトリクスの組み合わせから1行の天気メタファを生成
    - 例: "晴れ - 順調に進行中"、"曇り - focusが5日前、要アップデート"、"雷雨 - overdue 3件 + 未解決tension 5件"
  - 表示: カード上部に天気アイコン + 1行テキスト

  What's Next が「次にやること」なのに対し、天気予報は「今の状態」のエモーショナルな可視化。ダッシュボードを開いた瞬間に
  どこがヤバいか直感的にわかる。

  ただしLLMコストを抑えるため、全プロジェクト分を1回のAPI呼び出しでバッチ処理するのがポイント。

  ---
  6. Git Commit → Focus Bridge (コミットからフォーカス更新)

  git の最近のコミットを読み取り、current_focus.md への反映を提案する機能。

  - トリガー: Editor の Update Focus ボタンの横に "From Git"
  ボタン追加、またはDashboardの未コミット件数クリック後のダイアログに追加
  - 入力データ: git log --oneline --since=<前回focus更新日> の結果 + 現在の current_focus.md
  - LLMの仕事: コミットメッセージ群から「何が進んで、何が残っているか」を読み取り、focus更新案を生成
  - 出力: FocusUpdateService と同じ承認フロー(diff表示 + refine)

  「コード書いたけど focus 更新するの忘れた」を救済する。Update Focus from Asana の Git 版。

  ┌───────────────────────────┬─────────────────────────────────────────┬──────────────┬────────┐
  │         アイデア          │               実装コスト                │ ユーザー価値 │ 面白さ │
  ├───────────────────────────┼─────────────────────────────────────────┼──────────────┼────────┤
  │ Context Switch Briefing   │ 中 (FocusUpdateService類似パターン)     │ 高           │ 中     │
  ├───────────────────────────┼─────────────────────────────────────────┼──────────────┼────────┤
  │ Tension Synthesizer       │ 中 (DecisionLogGeneratorServiceの逆)    │ 高           │ 中     │
  ├───────────────────────────┼─────────────────────────────────────────┼──────────────┼────────┤
  │ Weekly Digest             │ 中 (StandupGeneratorの拡張)             │ 高           │ 高     │
  ├───────────────────────────┼─────────────────────────────────────────┼──────────────┼────────┤
  │ Decision Impact Tracker   │ 高 (全decision_logのパース必要)         │ 中           │ 高     │
  ├───────────────────────────┼─────────────────────────────────────────┼──────────────┼────────┤
  │ Project Weather Report    │ 低 (1回のLLM呼び出し + カードUI追加)    │ 中           │ 高     │
  ├───────────────────────────┼─────────────────────────────────────────┼──────────────┼────────┤
  │ Git Commit → Focus Bridge │ 中 (git log取得 + 既存承認フロー再利用) │ 中           │ 中     │
  └───────────────────────────┴─────────────────────────────────────────┴──────────────┴────────┘

  個人的なおすすめは Weekly Digest と Context Switch Briefing です。前者は「書くのが面倒な振り返りが自動で出てくる」快感
  があり、後者は複数プロジェクト管理の最大の痛み(復帰コスト)を直撃します。

  3. Cross-Project Duplicate & Conflict Detector

  - 内容: 全プロジェクト横断で「同じ課題の多重起票」「decision logの方針衝突」を検知。
  - なぜ今: Linearは重複Issue検知をAIトリアージの中核に置いている。 (linear.app (https://linear.app/ai))
  - 挑戦点: 類似判定の誤検知（同名別件）を説明可能にすること。

  4. Daily/Weekly Ops Digest

  - 内容: Slack Recap/Linear Pulse型で、毎日・毎週の「進捗/ブロッカー/未解決」を1枚に圧縮して配信（アプリ内＋Markdown保
    存）。
  - なぜ今: “catch-up時間削減”ユースケースは定着済み。 (slack.com
    (https://slack.com/help/articles/25076892548883-Guide-to-AI-features-in-Slack))
  - 挑戦点: 重要イベントだけを残してノイズを削る要約設計。

  5. PR/Commit Impact to Context

  - 内容: Git差分からcurrent_focus.mdとdecision_logの更新候補を自動提示（「この変更は何を決めたか」）。
  - なぜ今: GitHubはPR要約・レビュー理解支援を標準化している。 (docs.github.com

  (https://docs.github.com/en/enterprise-cloud%40latest/copilot/how-tos/use-copilot-for-common-tasks/create-a-pr-summary))
  - 挑戦点: コード変更と意思決定の因果を過剰推論しないこと。

  6. AI Knowledge Gap Radar

  - 内容: 「探し物時間」を減らすため、よく参照されるのに古いファイルを自動検知して更新提案。
  - なぜ今: 情報探索ロスの削減は組織課題として明確。 (atlassian.com
    (https://www.atlassian.com/blog/state-of-teams-2025))
  - 挑戦点: “古いが正しい文書”を誤って更新対象にしない判定。

  次に着手するなら、効果と実装難易度のバランスで 1 → 4 → 3 が最適です。
